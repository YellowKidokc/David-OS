# File Sidecar Contract

The TOP AI FIS file identity layer uses sidecars instead of embedded scripts.

## Why

Sidecars let files and folders carry memory without changing the original file.
They are easy to back up, easy to inspect, and safe to sync.

## File Sidecars

For a file:

```text
example.pdf
example.pdf.fmeta
example.pdf.chi
example.pdf.fmeta.chi
```

### `.fmeta`

Plain text metadata generated by the labeler.

Required fields:

```text
# file meta
name:
path:
ext:
size:
md5:
created:
modified:
accessed:
tags:
category:
tier:
```

Future fields:

```text
file_id:
source_node_id:
bucket_id:
visibility:
allowed_agents:
last_indexed:
rename_style:
status:
```

### `.chi`

Classification / intelligence sidecar. It can describe meaning, category,
claims, concepts, or AI-generated classifications.

### `.fmeta.chi`

Classification specifically about the metadata sidecar.

## Folder Sidecars

For a folder:

```text
SomeFolder/
  .folder.fmeta
```

Required fields:

```text
# folder meta
folder_id:
name:
current_path:
root:
relative_path:
first_scanned:
last_scanned:
moved_since_last_scan:
previous_paths:
fingerprint:
child_folder_count:
file_count:
total_file_bytes:
extensions:
tags:
category:
notes:
```

## Protected Extensions

The hub and scanners should not treat these as junk:

```text
.fmeta
.chi
.fisnote
.fistag
.fisdead
```

## Execution Rule

No executable code goes inside normal files. The sidecars are metadata only.
Workers read sidecars and the hub decides what to do.
